# OpenList 4.2.2 后续改动同步设计

## 目标与范围

将 PowerList 当前已同步的上游基线 `3b1760e9` 之后，到 `openlist/main` 当前提交 `eb486712` 为止的 OpenList 改动合入本仓库。`3b1760e9` 已包含在 PowerList 的上一轮 squash 同步中，因此不重复应用。

同步包含上游业务代码、测试、依赖清单和普通文档的变更。PowerList 自有驱动、缓存、索引及播放逻辑继续保留；发生冲突时，在保留本仓库定制行为的基础上融合上游修复和功能。

## 合并方式

以 `3b1760e9..eb486712` 的累计差异为输入，形成一个 PowerList 同步提交。相比逐提交 cherry-pick，这与仓库上一轮上游同步的 squash 方式一致，也避免在约 75 个提交中反复处理同一区域冲突。

使用三方合并信息处理累计补丁。对自动合并失败的文件逐一检查共同祖先、本仓库版本和上游版本，不以整文件覆盖方式丢弃任一侧有效功能。

## 保护范围

以下内容不得因本次同步发生变化：

- `.github/workflows/**`
- `build.sh`
- `build/**`
- 根目录及子目录中的 `Dockerfile*`
- 其他仅用于构建、打包或发布的脚本；若在上游差异中发现，加入保护清单后再继续

`go.mod` 和 `go.sum` 属于 Go 依赖清单，需要随上游业务代码同步，不归入上述构建脚本保护范围。

合并前记录所有已跟踪保护文件的对象 ID，合并后逐项比较，保证内容和文件存在状态不变。上游新增加的 GitHub Actions 或构建脚本也不引入。

## 冲突处理原则

1. PowerList 独有功能和本地修复优先保留。
2. 上游安全修复、崩溃修复、接口适配和新增驱动在不破坏本地功能的前提下合入。
3. 同一逻辑已由 PowerList 等价实现时，避免重复代码。
4. 依赖版本以能够编译全部合入代码为准，并通过 `go mod tidy` 检查一致性；不修改受保护的构建脚本来迁就依赖问题。
5. 用户现有未跟踪文件不纳入提交，也不删除或覆盖。

## 验证

完成合并后执行：

- 检查工作区和最终提交只包含预期文件。
- 对比保护文件合并前后的对象 ID，并确认没有新增上游工作流或构建脚本。
- 运行 `gofmt` 检查合入的 Go 文件。
- 运行 `go test ./...`；若完整测试受环境或既有问题阻塞，再运行相关包测试并明确报告未验证范围。
- 检查 `go mod tidy` 是否会产生额外差异，保证依赖清单一致。

最终提交采用单个上游同步提交，并在提交信息中记录同步范围和保护约束。
